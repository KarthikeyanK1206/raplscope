raplscope — Architecture & Interview Guide

A Linux system energy profiler in Go: reads Intel RAPL hardware counters from sysfs and reports joules and watts for a time window, or for exactly as long as a wrapped command runs — like /usr/bin/time, but for energy.

1The 30-second pitch

"I built raplscope, a command-line energy profiler for Linux. Modern Intel and AMD CPUs have hardware energy counters called RAPL that the kernel exposes through sysfs. raplscope reads those counters and answers a question time can't: not just how long did my program run, but how many joules did the machine burn while it ran. You can monitor live power draw, or wrap any command — sudo raplscope -- gzip -9 big.iso — and get total energy, average and peak power, with the child's stdio and exit code passed through exactly like /usr/bin/time does. It's pure Go standard library, a single static binary, and every line of measurement logic is unit-tested without needing root or RAPL hardware, because the sysfs path is injectable." 

$ sudo raplscope -- gzip -9 big.iso

command        gzip -9 big.iso
exit code      0
package-0      412.33 J
total energy   412.33 J
average power  28.71 W
peak power     34.02 W
elapsed        14.363 s
samples        14

2Background: what RAPL is

RAPL (Running Average Power Limit) is a feature of Intel CPUs since Sandy Bridge (2011) — and modern AMD Zen chips — originally built so the processor can limit its own power. A side effect is a set of cumulative energy counters maintained by hardware. The Linux powercap driver publishes them as plain text files: 

/sys/class/powercap/
├── intel-rapl:0/              ← one per CPU socket ("package")
│   ├── name                   "package-0"
│   ├── energy_uj              cumulative microjoules since boot, modulo wrap
│   ├── max_energy_range_uj    the counter's ceiling, e.g. 262143328850 (≈262 kJ)
│   ├── intel-rapl:0:0/        subdomain: "core"    (skipped — part of package)
│   └── intel-rapl:0:1/        subdomain: "uncore"  (skipped — part of package)
├── intel-rapl:1/              "psys" on some machines (skipped — includes packages)
└── intel-rapl-mmio:0/         mmio duplicate of package-0 (skipped — double counting)

Three hardware facts drive the whole design:

  • The counter is cumulative and it wraps. At max_energy_range_uj it rolls over to zero — roughly every 45 minutes on a busy ~100 W package. Any tool that ignores this silently reports garbage on long runs.
  • It is root-only. Since the PLATYPUS side-channel attack (CVE-2020-8694, 2020), the kernel restricts energy_uj to mode 0400. RAPL readings were precise enough to leak cryptographic keys — which is also a testament to their accuracy.
  • It is per-socket, not per-process. The hardware counts everything on the package. No software can change that, so raplscope doesn't pretend to attribute energy to a single process — it documents the limitation instead.

3High-level architecture

Four small components with strictly one-way data flow. No component knows about the one downstream of it beyond the value it hands over.

The two data types that cross component boundaries:

type Snapshot struct {          // produced by rapl.go, consumed by stats.go
    At time.Time                // carries Go's monotonic clock reading
    UJ []uint64                 // raw cumulative microjoules, one per domain
}

type Result struct {            // produced by stats.go, consumed by output.go
    Domains     []DomainResult  // per-package joules
    TotalJoules float64
    AvgWatts    float64
    PeakWatts   float64         // interval-averaged, documented as such
    ElapsedSec  float64
    Samples     int
    Command     string          // wrap mode only
    ExitCode    *int            // wrap mode only
}

5Data flow, both modes

Monitor mode — sudo raplscope -duration 10s

  1. Parse and validate flags (interval ≥ 10 ms; -duration and a command are mutually exclusive → exit 2).
  2. DiscoverReader(root) scans the powercap tree for top-level package-* domains, or fails with a specific error.
  3. Take the initial boundary snapshot (t₀, energy₀ per domain).
  4. Loop on a time.Ticker: read → Accumulator.Add → live line on stderr, optional CSV row.
  5. Stop when -duration elapses or on SIGINT/SIGTERM via signal.NotifyContext.
  6. Take the final boundary snapshot, render the Result as table or JSON on stdout. Exit 0.

Wrap mode — sudo raplscope -- stress-ng --cpu 4 -t 5

  1. flag.Args() non-empty → wrap mode (stdlib flag stops at the first non-flag argument; -- also works natively).
  2. Discover, take the initial snapshot, then exec.Command(...).Start() with inherited stdin/stdout/stderr.
  3. One goroutine waits on cmd.Wait(); the main loop selects over three channels: the ticker (sample), the signal channel (forward to child, keep waiting), and the wait result (done).
  4. Immediately after the child exits, take the final boundary snapshot — the measurement window is exactly the child's lifetime.
  5. Report to stderr (the child owns stdout) and exit with the child's exit code; 128+N if it died from signal N; 1 if it failed to start.

6Key design decisions — the section interviewers actually ask about

Path injection instead of an interface

DiscoverReader(root string) takes the powercap root path; production passes /sys/class/powercap, tests pass a fake tree in t.TempDir(). That one string parameter is the entire testing seam — no Source interface, no mocks.

Why: the boundary to fake isn't "an energy source", it's the filesystem, and the filesystem is already an abstraction. An interface with exactly one implementation is speculative generality: it adds indirection today for a second implementation that may never exist. If a GPU or battery source ever lands, the interface gets extracted then, shaped by two real implementations instead of one guess.

I/O and math never mix

rapl.go returns raw cumulative counters; every calculation — wrap correction, deltas, watts, peak — lives in stats.go as pure code operating on Snapshot values.

Why: the math is the part that can be subtly wrong, so it's the part that must be trivially testable. Accumulator tests feed hand-built snapshots with exact timestamps and assert exact joules — no hardware, no sleeping, no flakiness. It also means a bug report can be reproduced from a CSV of raw readings alone.

Boundary snapshots define totals; ticker samples are garnish

Total energy = final snapshot − initial snapshot (wrap-corrected). The per-interval samples in between exist only for peak power, live output and the CSV series.

Why: a wrapped command might run 50 ms while the sampling interval is 1 s — zero ticks fire, yet the total is still exact because the boundaries were read at start and exit. Summing ticker samples instead would systematically lose the partial interval at each end. Corollary: with fewer than two full intervals there's no meaningful peak, so peak falls back to the average rather than inventing precision.

Wraparound is corrected at every delta

func wrapDelta(prev, cur, maxRange uint64) uint64 {
    if cur >= prev {
        return cur - prev
    }
    return (maxRange - prev) + cur
}

Why: the counter rolls to zero at max_energy_range_uj — every ~45 minutes under load, so it will happen in real use. Because correction is applied per interval and the intervals are chained, the total stays correct across multiple wraps in one run, as long as any single interval spans at most one wrap (a 1 s interval vs a 45 min wrap period: five orders of magnitude of margin).

Actual monotonic elapsed time, never the nominal interval

Watts = ΔE / Δt where Δt comes from subtracting time.Time values — which in Go carries the monotonic clock — not from assuming "the ticker fired, so 1 s passed".

Why: tickers jitter under load (exactly when power measurement is interesting), and the wall clock can step (NTP, DST) mid-measurement. Using the monotonic reading makes the denominator honest. This is the kind of detail that separates a measurement tool from a toy.

Report stream follows the /usr/bin/time precedent

Monitor mode reports on stdout (live progress lines on stderr). Wrap mode reports on stderr, because the child owns stdout.

Why: sudo raplscope -- myprog | consumer must deliver the child's stdout to the consumer byte-for-byte, with the energy report off to the side — exactly how /usr/bin/time behaves. Same reasoning puts monitor-mode live lines on stderr so raplscope -json | jq parses cleanly.

Concurrency budget: three goroutines, no locks

The main loop, a channel delivering signals, and (wrap mode only) one goroutine whose only job is done <- cmd.Wait(). All coordination is a single select; the Accumulator is touched by exactly one goroutine, so it needs no mutex.

Why: every additional goroutine is a liability in a measurement tool — scheduling noise pollutes timing, and shared state invites races. The select loop makes the whole lifecycle (tick / signal / child-exit) readable in one screen.

Zero dependencies, one package, hard guardrails

Stdlib flag instead of cobra; text/tabwriter instead of a table library; no internal/ packages; no config files; no colors.

Why: the project's stated philosophy is that every abstraction must be justified by a problem that exists today. A 630-line tool with a CLI framework, DI container and plugin system says worse things about its author than a plain one. Zero dependencies also means zero supply-chain surface and a build that will still work in ten years.

7Testing strategy

The design constraint: every piece of logic must run in CI on any Linux runner — no root, no RAPL hardware. That's achieved with one seam and one discipline:

  • The seam: tests build a fake powercap tree in t.TempDir() (helper: writeFakeDomain(t, root, "intel-rapl:0", "package-0", energy, max)) and point DiscoverReader at it. The fake tree is generated at test time, not committed under testdata/ — the directory names contain :, which breaks git checkouts on Windows.
  • The discipline: because stats.go does no I/O, the math needs no seam at all — tests construct Snapshot values with exact timestamps and assert exact joules.

9Honest limitations

  • System-wide, not per-process. RAPL counts the whole socket. Wrapping a command measures system energy while it ran, background load included. This is a hardware property; tools claiming per-process joules from RAPL are modeling, not measuring.
  • Root required (post-PLATYPUS kernels).
  • Bare metal only — no VMs, most containers, WSL2, or ARM.
  • Peak is interval-averaged, not an instantaneous spike.
  • CPU package energy only — no GPU, DRAM domain, battery, or platform (psys) totals.

10Interview Q&A — click a question to reveal the answer (self-quiz)

What happens if the RAPL counter wraps twice between two of your samples?

The delta undercounts by one full max_energy_range_uj — wrapDelta can only detect a single wrap, because a cumulative counter modulo N loses information beyond one period. The defense is rate: the wrap period is ~45 minutes at full load and the maximum sampling interval anyone would sanely configure is seconds, so a double wrap within one interval would require power draw five orders of magnitude beyond the hardware's range. It's documented, not defended against with code — an example of matching engineering effort to physical reality.

Why did you not define a Source interface for energy readers?

Because there is exactly one source. An interface with one implementation is indirection with no consumer — you pay the abstraction cost today for a hypothetical tomorrow. The actual variability I need is "where is the sysfs tree", and a string parameter covers it for both production and tests. If a second source (GPU, battery) ever lands, I'd extract the interface then, and it would be shaped by two real implementations rather than one guess. Interfaces are cheap to add later in Go precisely because they're satisfied implicitly.

How do you test hardware-dependent code without the hardware?

The hardware interface is just files with integers in them, so tests generate a fake powercap tree in a temp directory — including decoys like psys, core subdomains and mmio duplicates that discovery must skip — and inject its path. The permission failure is simulated with chmod 0. Everything downstream of the reader is pure math on Snapshot values and needs no faking at all. The one thing this can't prove — that real kernels really lay files out this way — is covered by a documented manual smoke test on real hardware.

Walk me through what happens when I press Ctrl+C while wrapping a command.

The terminal sends SIGINT to the foreground process group — both raplscope and the child. raplscope has the signal routed to a channel via signal.Notify, so it doesn't die; its select loop forwards the signal to the child and keeps waiting. When the child exits, the Wait() goroutine delivers the status, raplscope takes the final energy snapshot, prints the report for the window that actually elapsed, and exits with 128+N — the shell convention for death-by-signal-N, the same thing /usr/bin/time does. The invariant: the child controls its own death; raplscope only observes and reports.

Why does total energy come from boundary snapshots instead of summing the per-second samples?

Because the counter is cumulative, the difference between the last and first readings is the total — the samples in between add no information to the sum, only resolution for peak power and the CSV curve. Summing samples would also break on short windows: a 50 ms wrapped command sees zero ticker fires, but its boundary snapshots are taken at Start() and right after Wait(), so the total is still exact. The general principle: derive totals from the most direct measurement available, use sampling only for what genuinely needs time resolution.

Why is average power computed with monotonic time? What could go wrong otherwise?

Two failure modes. First, ticker jitter: under load a "1 s" tick might arrive after 1.2 s; dividing by the nominal interval would overstate power by 20% exactly when the machine is busiest. Second, wall-clock steps: NTP adjustments or a suspended laptop can move time.Now() backwards or jump it forward, producing negative or absurd Δt. Go's time.Time carries a monotonic reading, and subtracting two of them uses it automatically — so Δt is always the physical duration between the two reads.

Why can't this tool tell me how much energy my process used?

Because the counter is implemented in the package hardware and counts every core, cache and memory controller on the socket, no matter which process was running. Attribution would require sampling per-process CPU time and dividing energy proportionally — a model with real error bars (shared caches, frequency scaling, interference), not a measurement. I chose to report the honest number — system energy during the window — and document the distinction, rather than print a per-process figure that looks precise and isn't. For comparisons (gzip vs zstd on an idle machine) the honest number is exactly what you want anyway.

Why is energy_uj root-only, and why do you mention a CVE in an error message?

PLATYPUS (CVE-2020-8694, 2020) showed RAPL readings are precise enough to act as a power side channel — attackers could recover cryptographic keys by correlating energy with instruction data. The kernel's fix made energy_uj mode 0400. The error message names the CVE because "permission denied" alone sends a user to chmod or bug trackers; telling them why it's restricted and that sudo is the intended path turns a dead end into a one-step fix. Error messages are UI.

Why did you skip the psys domain? Isn't more data better?

psys covers the whole platform including the packages, so summing psys with package-0 counts the CPU twice. Reporting psys alone was an option, but it's not present on most machines and its coverage varies by platform, which would make numbers incomparable across systems. Summing only top-level package-* domains — while also excluding core/uncore subdomains (subsets of package) and intel-rapl-mmio (a second interface to the same counters) — is the one choice that never double counts. Correctness of the sum beat completeness of the sources.

Describe the concurrency model. Where are the locks?

There are no locks, by construction. At most three goroutines exist: the main loop, the runtime delivering signals to a channel, and in wrap mode one goroutine doing done <- cmd.Wait(). All events — ticker fire, signal arrival, child exit — converge on one select in one goroutine, and that goroutine is the only one touching the Accumulator. Serializing events through channels instead of sharing state is the idiomatic Go answer, and in a measurement tool it has a second payoff: fewer goroutines means less scheduler noise in the timings.

How would you add GPU power measurement?

First establish that the semantics differ: NVML gives instantaneous milliwatts, not a cumulative energy counter, so it needs integration over samples rather than boundary subtraction — which means it can't just masquerade as another RAPL domain. I'd extract a small interface shaped by what the accumulator actually needs (probably "give me energy since last call"), implement it twice — the RAPL reader via counter deltas, the GPU source by integrating power samples — and report them as separate domains, never summed silently with CPU joules since they're measured differently. The key point: I'd let the second real implementation shape the abstraction, which is exactly why it doesn't exist yet.

Why stdlib flag instead of cobra? Why zero dependencies at all?

The tool has seven flags and two modes; flag covers that in ~20 lines including a custom usage string. Cobra would add subcommand machinery, completions and ~10 transitive dependencies to serve needs this tool doesn't have. Zero dependencies buys concrete things: no supply-chain exposure, no version churn, go build works offline forever, and the binary is fully auditable by reading one directory. The general rule I followed: every dependency and every abstraction must be justified by a problem that exists today, not a problem I can imagine.

What was the hardest bug or subtlety you hit?

Two honest candidates. One: the final boundary snapshot creates a partial interval — a run of 4.3 s at a 1 s interval ends with a 0.3 s tail — and feeding that tail into peak-power tracking produced noise spikes, since tiny Δt amplifies quantization error. Fix: Finish() folds the tail into the totals but excludes it from peak. Two: in testing, my fake-counter simulator wrote files with truncate-then-write, and the tool occasionally read an empty file mid-write. Real sysfs reads are atomic (the kernel generates content at read time), so the right fix was making the simulator write atomically via rename — a good lesson in fixing the model, not adding retry logic to production code for a failure mode that can't occur there.

If fewer than two sampling intervals fit in the window, what does peak power mean?

Nothing — so the tool doesn't report a fake one. Peak is defined as the maximum over full-interval average powers; with zero or one full interval there's no population to take a maximum over, so PeakWatts falls back to AvgWatts and the README documents it. The alternative — reporting the single interval or the partial tail as "peak" — would be precision theater. Small case, but it's the pattern throughout: never let the output imply more measurement than actually happened.
